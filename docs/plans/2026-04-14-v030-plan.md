## Phase 1: Webhook Completion (PR 1)

### Task 1: Dispatch `message.queued` webhook event

**Files:**
- Modify: `services/chorus-server/src/queue/enqueue.rs`

**Steps:**
1. After message is inserted to DB and pushed to Redis queue, call `dispatch_webhooks()` with `message.queued` event
2. Build `WebhookPayload` with status `"queued"`, no provider yet
3. Import `webhook_dispatch::dispatch_webhooks` and `WebhookPayload`

**Verify:** `cargo check -p chorus-server`

### Task 2: Dispatch `message.sent` webhook event

**Files:**
- Modify: `services/chorus-server/src/queue/worker.rs`

**Steps:**
1. After provider.send() succeeds but before updating status to `"delivered"`, dispatch `message.sent`
2. This goes between the send call and the DB update (around line 140-150)
3. Payload includes provider name, status `"sent"`

**Verify:** `cargo check -p chorus-server`

### Task 3: Add webhook integration tests

**Files:**
- Modify: `services/chorus-server/tests/api_test.rs`

**Steps:**
1. Add tests for webhook CRUD: create (201 + secret), list (secret redacted), delete (204)
2. Add test for invalid event name (400)
3. Add test for webhook signature verification logic (unit test in `webhook_dispatch.rs`)

**Verify:** `cargo test -p chorus-server`

### Task 4: Lint + create PR

```bash
cargo fmt --all && cargo clippy --workspace -- -D warnings && cargo test --workspace
```

**Steps:**

### Task 6: Implement metrics collection

**Files:**
- Create: `services/chorus-server/src/metrics.rs`
- Modify: `services/chorus-server/src/app.rs`
- Modify: `services/chorus-server/src/main.rs`

**Metrics to track:**
- `chorus_http_requests_total` (counter, labels: method, path, status)
- `chorus_http_request_duration_seconds` (histogram, labels: method, path)
- `chorus_messages_total` (counter, labels: channel, status, provider)
- `chorus_queue_depth` (gauge)
- `chorus_webhook_deliveries_total` (counter, labels: event, status)
- `chorus_provider_errors_total` (counter, labels: provider)

**Steps:**
1. Create `metrics.rs` with `setup_metrics()` function
2. Add `/metrics` endpoint in router (no auth required)
3. Add metrics recording in worker (message counts, provider errors)
4. Add metrics recording in webhook dispatch (delivery counts)

### Task 7: Add structured log formatting for Loki

**Steps:**
1. Add `tracing-subscriber` JSON layer for production (`CHORUS_LOG_FORMAT=json`)
2. Keep human-readable format as default for dev
3. Add span fields: `account_id`, `message_id`, `provider` for correlation

### Task 8: Enhance health endpoint

## Phase 3: Stripe Billing (PR 3)

### Task 10: Add billing database migrations

**Files:**
- Create: `services/chorus-server/src/db/migrations/004_create_billing.sql`

**Tables:**
- `billing_plans` (id, slug, name, price_cents, sms_quota, email_quota)
- `subscriptions` (id, account_id, stripe_customer_id, stripe_subscription_id, plan_id, status, period_start/end)
- `usage` (id, account_id, period_start/end, sms_sent, email_sent, sms_cost_microdollars, email_cost_microdollars)

### Task 11: Add Stripe dependency and billing module

**Files:**
- Modify: `services/chorus-server/Cargo.toml`
- Create: `services/chorus-server/src/billing/mod.rs`
- Create: `services/chorus-server/src/billing/stripe_client.rs`
- Create: `services/chorus-server/src/billing/usage.rs`

**Steps:**
1. Add `stripe` crate dependency
2. Create `StripeClient` wrapper (customer, subscription, checkout session)
3. Create `UsageTracker` that aggregates message counts per billing period
4. Increment usage counters in worker after successful send

### Task 12: Add billing repository

**Files:**
- Create: `services/chorus-server/src/db/billing.rs`
- Modify: `services/chorus-server/src/db/mod.rs`

**Steps:**
1. Implement `BillingRepository` trait (plans, subscriptions, usage CRUD)
2. Implement `PgBillingRepository`
3. Add quota check before enqueue (reject if over quota)

### Task 13: Add billing API routes

**Files:**
- Create: `services/chorus-server/src/routes/billing.rs`
- Modify: `services/chorus-server/src/app.rs`

**Endpoints:**
- `GET /v1/billing/plan` — Current plan + usage
- `GET /v1/billing/plans` — Available plans
- `POST /v1/billing/checkout` — Create Stripe checkout session
- `POST /v1/billing/cancel` — Cancel at period end
- `GET /v1/billing/usage` — Current period usage
- `GET /v1/billing/invoices` — Invoice history
- `POST /v1/billing/webhook/stripe` — Stripe webhook receiver (no auth, verify Stripe signature)
